## 1. Goal & Scope

### Goal
Add **profitability** and **pricing** as fully-functional consulting types alongside the existing `market_entry` pipeline. Refactor first so all three types share one engine. The `ma` task continues to use its V2 placeholder stub (no change in V1.5).

### In scope
- Extract a generic consulting engine from `app/agents/market_entry/` into a new `app/agents/_engine/` package.
- Define a `ConsultingProfile` dataclass that describes the framing prompt, ordered stages (each with its own prompt), synthesis prompt, and audit checklist.
- Port `market_entry` to be defined as a `ConsultingProfile` with **zero behavioral change** (all existing tests pass unchanged).
- Add `profitability` and `pricing` profiles, each with their own prompts, framing questions, and stage definitions.
- Alembic migration enabling both new task types in the catalog.
- Worker dispatches by `task_id` via a `PROFILE_REGISTRY` lookup.
- New end-to-end smoke tests per type using `FakeChatModel`.

### Explicitly out of scope
- The full `ma` (M&A) pipeline — remains the V2 placeholder stub.
- Frontend changes. The picker on `/` already reads `/tasks` and disables un-enabled types; the 4-pane run workspace is task-agnostic. New types appear automatically.
- New stages, new tools, or new providers beyond what V1 already wires.
- Per-type artifact paths — all three types write `final_report.md` (matches `ma` and existing `ReportView` contract).
- Per-type frontend icons/taglines/example-goals (deferred to V1.6 or later).
- Reviewer behavior changes beyond a generic-wording rephrase (no new gates).

### Worker dispatch

In `app/api/runs.py::create_run` (replaces today's branch):

```python
if run.task_id == "ma":
    TASK_REGISTRY.spawn(f"run:{run.id}", run_ma_stub(run.id))
else:
    profile = PROFILE_REGISTRY.get(run.task_id)
    if profile is None:
        raise HTTPException(400, f"Unsupported task_type: {run.task_id!r}")
    model_factory = await factory_builder(run.id)
    TASK_REGISTRY.spawn(
        f"run:{run.id}",
        start_framing(run.id, profile=profile, model_factory=model_factory),
    )
```

`start_framing` and `continue_after_framing` (in `app/workers/run_worker.py`) gain a `profile: ConsultingProfile` parameter and pass it to `build_consulting_graph(profile, ...)`. The graph builder's signature changes from market-entry-specific to profile-driven, but its return type (`CompiledGraph`) and downstream usage are identical.

### Why this shape
- **Generic engine + profile registry** keeps the pipeline mechanics (LangGraph wiring, reviewer loop, evidence/citation handling, budget tracker hookup) in one place. Bug fixes and feature upgrades benefit all three types automatically.
- **Profiles as Python dataclasses** give type checking, IDE refactor support, and let prompts live as multiline `.md` files (not awkward YAML/JSON heredocs).
- **A registry dict** rather than dynamic discovery means missing prompt files surface at module import time (process startup), not mid-run.

## 3. Data Shapes

```python
# app/agents/_engine/profile.py
from __future__ import annotations
import importlib.resources
from dataclasses import dataclass

@dataclass(frozen=True)
class ProfileStage:
    """One research stage in a consulting profile."""
    id: str                        # "stage_1", "stage_2", … (stable across profiles)
    name: str                      # "Revenue diagnosis"
    prompt_path: str               # "stages/1.md" — relative to profile.prompts_package
    web_search_required: bool      # if True, prompt is told it must produce citations
    required: bool = True          # reviewer can mark optional stages "skip"

@dataclass(frozen=True)
class ConsultingProfile:
    slug: str                      # matches task_types.slug
    display_name: str              # for logs/traces only
    prompts_package: str           # e.g. "app.agents.market_entry.prompts"
    framing_prompt_path: str       # e.g. "framing.md"
    stages: tuple[ProfileStage, ...]
    synthesis_prompt_path: str     # e.g. "synthesis.md"
    audit_prompt_path: str         # e.g. "audit.md"

    def load_prompt(self, rel_path: str) -> str:
        """Reads a prompt file shipped with the profile package.
        Raises FileNotFoundError at registry-import time if the file is missing."""
        return (
            importlib.resources.files(self.prompts_package)
            .joinpath(rel_path)
            .read_text(encoding="utf-8")
        )
```

```python
# app/agents/_engine/registry.py
from app.agents._engine.profile import ConsultingProfile
from app.agents.market_entry.profile import MARKET_ENTRY_PROFILE
from app.agents.profitability.profile import PROFITABILITY_PROFILE
from app.agents.pricing.profile import PRICING_PROFILE

PROFILE_REGISTRY: dict[str, ConsultingProfile] = {
    p.slug: p for p in (MARKET_ENTRY_PROFILE, PROFITABILITY_PROFILE, PRICING_PROFILE)
}
```

The example above shows the **final V1.5 end state**. The registry grows incrementally per commit: M9.1 contains only `MARKET_ENTRY_PROFILE`; M9.2 adds `PROFITABILITY_PROFILE`; M9.3 adds `PRICING_PROFILE`. Imports for not-yet-shipped profiles are added in the same commit that ships those profiles.

The registry is built once at import time. Each profile's prompts are validated by calling `load_prompt` for `framing_prompt_path`, every `stage.prompt_path`, `synthesis_prompt_path`, and `audit_prompt_path` from a `_validate()` helper invoked at the bottom of `registry.py`. Missing files crash the worker process at boot, not mid-run.

`RunState` (today in `market_entry/state.py`, moved to `_engine/state.py`) stays unchanged. Per-type variation lives in the profile, not in state.

## 4. Stage Definitions

### Profitability — root-cause-tree on margin

| # | Stage `id` | Name | What it produces | `web_search_required` |
|---|---|---|---|---|
| 1 | `stage_1` | Revenue diagnosis | Volume / price / mix decomposition; identifies which lever moved | `false` |
| 2 | `stage_2` | Cost diagnosis | Fixed vs. variable split; cost-per-unit drivers; trend vs. baseline; benchmarks | `true` |
| 3 | `stage_3` | Segmentation | Profitability split by product / customer / channel / geography | `true` |
| 4 | `stage_4` | Root-cause hypothesis | Ranks the 1–2 biggest drivers with quantified impact | `false` |
| 5 | `stage_5` | Levers | Actionable recommendations with expected $ impact + effort estimate | `false` |

**Framing questions:** time horizon of the decline, business unit / product scope, available financial data sources (P&L, COGS breakdown, customer cohorts), key stakeholders.

### Pricing — set or reset price for a product/service

| # | Stage `id` | Name | What it produces | `web_search_required` |
|---|---|---|---|---|
| 1 | `stage_1` | Cost baseline | Unit economics; gross-margin floor; cost-plus reference price | `false` |
| 2 | `stage_2` | Customer value | Willingness-to-pay analysis; segments; substitutes; value-based reference price | `true` |
| 3 | `stage_3` | Competitive landscape | Comparable prices; positioning map; price–feature curve | `true` |
| 4 | `stage_4` | Pricing model | Cost-plus vs. value-based vs. tiered vs. dynamic — pick + rationale | `false` |
| 5 | `stage_5` | Recommendation | Price points per segment + rollout / communication plan | `false` |

**Framing questions:** product/service in scope, target customer segment, competitive context, current pricing (if any), launch vs. repricing, business constraint (margin floor, volume target, brand positioning).

### Why 5 stages
Mirrors `market_entry`. Reviewer logic, budget assumptions, and stage-numbering UI conventions all carry over unchanged.

## 5. Migrations

Two migrations, one per type, so each milestone commit ships consistent catalog-state and registry-state. (Shipping both rows in M9.2 would crash worker boot — registry validation would fail to load `pricing` prompts that don't exist yet.)

### `0008_enable_profitability.py` (lands in M9.2)

```python
def upgrade() -> None:
    op.execute(sa.text("""
        INSERT INTO task_types (slug, name, description, enabled) VALUES
            ('profitability', 'Profitability',
             'Diagnose declining margin: revenue/cost decomposition, segmentation, levers.', true)
        ON CONFLICT (slug) DO UPDATE SET
            name = EXCLUDED.name,
            description = EXCLUDED.description,
            enabled = EXCLUDED.enabled
    """))

def downgrade() -> None:
    op.execute(sa.text("DELETE FROM task_types WHERE slug = 'profitability'"))
```

### `0009_enable_pricing.py` (lands in M9.3)

```python
def upgrade() -> None:
    op.execute(sa.text("""
        INSERT INTO task_types (slug, name, description, enabled) VALUES
            ('pricing', 'Pricing',
             'Set or reset price: cost baseline, customer value, competitive scan, recommendation.', true)
        ON CONFLICT (slug) DO UPDATE SET
            name = EXCLUDED.name,
            description = EXCLUDED.description,
            enabled = EXCLUDED.enabled
    """))

def downgrade() -> None:
    op.execute(sa.text("DELETE FROM task_types WHERE slug = 'pricing'"))
```

`ON CONFLICT … DO UPDATE` makes each migration idempotent and lets us refine name/description copy by re-running. Foreign-key constraint from `runs.task_id` is `RESTRICT`, so `downgrade` will fail loudly if any runs reference these slugs (intended; protects historical data).

## 8. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Prompt-loading path breakage when files move from `market_entry/prompts/` (no actual move) or are referenced by relative path | `importlib.resources` keyed by `prompts_package` string in profile; validated at registry-import time, not run time. |
| Stage state shape diverges per type | One `RunState` for all profiles; per-type info lives in the profile, not state. If a future type needs new state, that's a V1.6 concern (extend `RunState` with optional fields, all profiles still satisfy it). |
| Reviewer prompt has market-entry-specific wording (e.g., "Is this market analysis sufficient?") | Audit `_engine/nodes/reviewer.py` (the prompt template specifically) and replace market-specific terms with generic language ("Is this stage's analysis sufficient to support the next stage?"). Verify market_entry tests still pass — the assertions check structure, not wording. |
| Token cost of three 5-stage pipelines on long docs | Already tracked by M7.1's `BudgetTracker`; UsagePanel shows running cost. Acceptable for V1.5. No hard cap. |
| Pricing/profitability prompts are first-draft quality | Acknowledged. V1.6 iterates based on real-run feedback. Tests assert structural correctness (stage sections present, citations present) not content quality. |
| `runs.task_id` foreign key blocks `downgrade` of `0008` / `0009` | Intended. `RESTRICT` protects historical runs. Downgrade requires manual cleanup of dependent rows — this is a feature, not a bug. |
| Profile registry validation crashes worker boot if a single prompt file is missing | Intended fail-fast. Adding a CI check (one of the existing tests imports the registry) catches this before deploy. |
| `runner.py` per type is dead code if worker dispatches via profile lookup | Acceptable. The per-type `runner.py` is one or two lines and serves as a discoverable entry point for ad-hoc scripts and future per-type CLI commands. Cost: ~3 LOC per type. |

## 9. Acceptance Criteria

V1.5 is complete when **all** of:

1. `bash scripts/check.sh` (with `CHECK_BUILD=1`) exits green.
2. `pytest` reports ≥ 132 passing (129 existing + at least 3 new: 1 unit profile test + 2 per-type smoke tests).
3. `alembic upgrade head` reaches `0009_enable_pricing` cleanly on a fresh DB.
4. `GET /tasks` returns 4 entries — `market_entry`, `profitability`, `pricing` enabled; `ma` enabled (stub).
5. Manually creating a run with `task_type="profitability"` and `task_type="pricing"` from the picker UI completes end-to-end against real LLMs (smoke-tested by the dev, not gated in CI).
6. Three commits land on `main` in order: M9.1 (refactor), M9.2 (profitability), M9.3 (pricing). Each is a self-contained green checkpoint.
